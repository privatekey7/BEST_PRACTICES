# Best Practices Guide

Универсальный full-stack набор практик написания кода — от идеи до продакшена.
Подходит для любого проекта: backend-сервис, фронтенд, desktop-приложение,
мобильное приложение или библиотека. Стек и инструменты — актуальные на **июнь 2026**.

> Этот файл задуман как переносимый стандарт: копируй его в корень нового
> проекта и адаптируй под конкретный тип приложения по тегам применимости.

---

## Как пользоваться этим документом

Документ разбит на три части:

- **Часть I — База.** Применимо к **любому** проекту. Читать всем и всегда.
- **Часть II — Специфика по типу проекта.** Разделы помечены тегом — бери
  только то, что относится к твоему типу приложения.
- **Часть III — Актуальный стек 2026.** Справочник: какие инструменты выбирать
  сегодня для каждой экосистемы.

### Легенда тегов применимости

| Тег | Тип проекта |
|-----|-------------|
| ⚙️ **universal** | любой проект |
| 🌐 **server** | backend / API / сетевой сервис / контейнер |
| 🎨 **frontend** | веб-приложение в браузере (SPA/SSR) |
| 🖥 **desktop** | десктоп-приложение (Tauri/Electron/нативное/PyInstaller) |
| 📱 **mobile** | мобильное приложение (iOS/Android) |
| 📦 **library** | библиотека / пакет / SDK |
| 🤖 **ai-agent** | LLM-агент / AI-продукт в продакшене (агент = сам продукт) |

Правило №1: **не применяй серверные практики к desktop/CLI и наоборот.** Самая
частая ошибка новичка — например, логировать desktop-приложение «только в
stdout» (его там нет) или ставить health-check эндпоинты в библиотеку.

# Часть I. База (для любого проекта) ⚙️

## 1. Процесс разработки

### 1.1. От задачи до релиза

```
Задача (issue/ticket)
  → Уточнение требований
  → Создание ветки
  → Код + локальный прогон линтера/тестов
  → Коммит (Conventional Commits)
  → Push + Pull Request
  → CI (зелёный) + Code review
  → Squash-merge в main
  → CI/CD → релиз/деплой
```

### 1.2. Одна задача — один PR

`fix бага + новая фича + рефакторинг` в одном PR невозможно нормально ревьюить,
откатывать и искать в нём причину регрессии.
Правило: **один PR = одно логическое изменение.** Держи PR < ~400 строк диффа.

### 1.3. Feature flags (флаги включения)

Новый функционал — **за флагом**, а не сразу всем. Можно выкатить выключенным
(тесты идут, юзеры не видят), включить для canary-группы, мгновенно выключить
(kill switch).

```ts
// config.ts
export const FEATURES = {
  newUi: false,            // скоро включим
  darkMode: true,          // уже работает
  experimentalEngine: false, // ещё тестируем
} as const;

export const isEnabled = (f: keyof typeof FEATURES) => FEATURES[f];
```

Для команд побольше — вынеси флаги во внешний сервис (Unleash, Flagsmith,
OpenFeature-совместимый), чтобы переключать без передеплоя.

### 1.4. Code review

Каждый PR ревьювит другой человек (или, как минимум, ты сам спустя «холодную
голову»). Что проверять:

- **Логика** — правильно ли решает задачу, краевые случаи.
- **Безопасность** — инъекции, утечки данных, проверка прав.
- **Производительность** — нет ли O(n²), N+1-запросов, лишних ре-рендеров.
- **Читаемость** — поймёт ли другой через полгода.
- **Тесты** — покрыт ли новый код.

## 2. Git и работа с репозиторием

### 2.1. Conventional Commits ⚙️

Строгий формат сообщений → автоматический changelog, авто-semver, удобный поиск.

```
<type>(<scope>): <описание в повелительном наклонении>

[тело: что и зачем, а не как]

[BREAKING CHANGE: ...]
```

| Тип | Когда |
|-----|-------|
| `feat` | новый функционал (→ minor) |
| `fix` | исправление бага (→ patch) |
| `refactor` | рефакторинг без изменения поведения |
| `perf` | оптимизация производительности |
| `test` | тесты |
| `docs` | документация |
| `build` | сборка, зависимости |
| `ci` | пайплайны CI |
| `chore` | обслуживание |
| `style` | форматирование (не CSS) |

`feat!:` или `BREAKING CHANGE:` в теле → major-версия.

```
feat(auth): add password reset flow
fix(parser): handle empty input on punctuation
refactor(config): extract hotkey validation
perf(list): memoize row renderer to cut re-renders
```

### 2.2. Ветвление — Trunk-Based Development

Рекомендуется для большинства команд (особенно соло и небольших):

- `main` — всегда стабильная, готовая к релизу.
- Короткоживущие ветки `feat/<краткое-описание>`, `fix/<issue-id>`.
- Ветка живёт часы/дни, не недели. После merge — удаляется.
- В main попадают только через PR с зелёным CI. Прямой push в main запрещён
  (защити branch protection rules).

GitFlow (`develop` + `release/*`) сегодня избыточен для большинства — оставь его
только для софта с поддержкой нескольких параллельных версий.

### 2.3. `.gitignore` и что НИКОГДА не коммитить

В `.gitignore`: артефакты сборки (`dist/`, `build/`, `*.exe`, `target/`),
кэши (`__pycache__/`, `.pytest_cache/`, `node_modules/`, `.venv/`),
секреты (`.env`, `.env.local`, `*.key`, `*.pem`), локальные IDE-конфиги.

**Никогда не коммить:** пароли/токены/API-ключи/сертификаты, дампы БД,
большие бинарники (>10 МБ → Git LFS или внешнее хранилище), личные конфиги.

Если секрет всё же попал в историю — он **скомпрометирован**: ротируй ключ,
затем чисти историю (`git filter-repo`). Подключи секрет-сканер в CI
(gitleaks / trufflehog) и pre-commit, чтобы это не повторялось.

### 2.4. Git Worktrees — параллельная работа без stash

`git worktree` позволяет держать несколько рабочих директорий одного
репозитория одновременно — каждая на своей ветке, с общей историей.

```bash
# Создать worktree для срочного фикса, не бросая текущую фичу
git worktree add ../myproject-hotfix fix/critical-bug

# Поработать, замержить, удалить
cd ../myproject-hotfix
# ... fix, commit, push, merge ...
cd ../myproject
git worktree remove ../myproject-hotfix
```

Когда полезно:

- **Срочный фикс** посреди незакоммиченной фичи — не нужен `stash`.
- **Параллельные AI-агенты:** каждый агент работает в своём worktree, не
  конфликтуя с другими (подробнее — §19).
- **Code review:** открыл PR-ветку в отдельной директории и смотришь, пока
  основная работа продолжается.

## 3. Написание кода

### 3.1. Читаемость > краткость ⚙️

```python
# ПЛОХО — приходится расшифровывать
def proc(d):
    r = {}
    for k, v in d.items():
        if v > 10:
            r[k.upper()] = v * 2
    return r

# ХОРОШО — понятно без комментариев
def normalize_scores(raw_scores: dict[str, int]) -> dict[str, int]:
    result: dict[str, int] = {}
    for key, value in raw_scores.items():
        if value > THRESHOLD:
            result[key.upper()] = value * 2
    return result
```

Комментируй **почему**, а не **что**: код уже говорит, что он делает; комментарий
объясняет причину неочевидного решения.

### 3.2. Имена

- **Функции — глаголы:** `getUser()`, `saveConfig()`, `validateEmail()`.
- **Переменные — существительные:** `userCount`, `configPath`, `activeModel`.
- **Булевы — вопрос:** `isInstalled`, `hasPermission`, `isEnabled`.
- **Коллекции — множественное число:** `users`, `modelIds`, `errors`.
- **Константы — UPPER_CASE:** `DEFAULT_HOTKEY`, `SAMPLE_RATE`, `MAX_RETRIES`.

### 3.3. Принцип единственной ответственности (SRP)

Функция делает **одну вещь**. Если в названии есть «и» — скорее всего, надо
разбивать.

```python
# ХОРОШО — три простые функции вместо одной «делающей всё»
def read_json(path: Path) -> dict: ...
def extract_active_names(data: list[dict]) -> list[str]: ...
def write_lines(path: Path, lines: list[str]) -> None: ...
```

### 3.4. DRY и магические числа

Повторяется дважды → выноси. Любое «магическое» число/строку → именованная
константа в одном месте.

```python
MIN_RECORD_MS = 250
VOLUME_SCALE = 0.5

if duration_ms < MIN_RECORD_MS:
    return
volume = raw * VOLUME_SCALE
```

Но не доводи до абсурда (AHA: «Avoid Hasty Abstractions») — преждевременная
абстракция ради одного будущего случая хуже разумного дубля.

### 3.5. Обработка ошибок

- Проверяй краевые случаи (пусто, `None`/`null`, нет файла, таймаут).
- Лови **конкретные** исключения, не «глотай» всё подряд.
- Сообщения об ошибках — осмысленные и с контекстом.

```python
try:
    data = load_config()
except FileNotFoundError:
    data = {}  # нормально: файла ещё нет
except json.JSONDecodeError:
    logger.warning("Битый конфиг, начинаю с чистого листа")
    data = {}
except OSError as e:
    raise RuntimeError(f"Не могу прочитать конфиг: {e}") from e
```

В TS/JS — не глотай промисы: всегда `await` + `try/catch` или `.catch()`;
включи `no-floating-promises`.

### 3.6. Типизация ⚙️

Типы — везде, где можно. Это бесплатная документация + ловля багов type
checker'ом + безопасный рефакторинг.

```python
# Python 3.10+: используем X | None, НЕ Optional[X]
def find_user(user_id: int) -> User | None:
    """Поиск пользователя по ID. Возвращает None, если не найден."""
    ...
```

```ts
// TS: strict-режим обязателен. Избегай any → используй unknown + сужение.
function findUser(id: number): User | null { ... }
```

Для данных «с границы» (API, env, файлы) — валидируй схемой (Pydantic v2 в
Python, Zod/Valibot в TS), а не доверяй типам на слово.

### 3.7. Настройки — отдельно от логики

Никогда не зашивай пути, порты, URL, ключи в тело функции. Конфиг — из env /
файла / централизованного модуля.

```python
# ПЛОХО
path = "C:\\Users\\user\\AppData\\app\\models"

# ХОРОШО
def get_models_dir() -> Path:
    return appdata_root() / "models"
```

### 3.8. Иммутабельность и чистые функции

Предпочитай чистые функции (результат зависит только от входа, без побочных
эффектов) — их легко тестировать и переиспользовать. Не мутируй входные
аргументы; возвращай новые значения. В TS — `const`, `readonly`, `as const`.

### 3.9. SOLID — принципы, а не ритуал ⚙️

SOLID часто пишут в резюме, но применяют наизусть и не к месту. Это пять
ориентиров, а не обязательный обряд:

- **S — Single Responsibility:** у класса/модуля одна причина для изменения (§3.3).
- **O — Open/Closed:** расширяй поведение, не переписывая существующий код.
- **L — Liskov Substitution:** наследник не ломает контракт базового типа.
- **I — Interface Segregation:** узкие интерфейсы вместо одного «толстого».
- **D — Dependency Inversion:** завязывайся на абстракции, а не на детали.

**Типичная ошибка** — «SOLID ради SOLID»: интерфейс на каждый класс, фабрики и
слои абстракции там, где хватило бы функции. Это та же преждевременная абстракция
(§3.4). Принцип нужен, когда снимает реальную боль, а не для галочки.

### 3.10. Размер функции — ориентир, не догма

«Сколько строк должно быть в функции» — вечный холивар, и **чем проще вопрос, тем
яростнее спор**. Жёсткого числа нет, есть ориентиры:

- Функция читается целиком без скролла и держится в голове.
- Делает одну вещь (§3.3) и держит один уровень абстракции.
- Дробление ради «5 строк на функцию» плодит лишние сущности — это тоже плохо.

Контекст важнее цитат из Clean Code: одни и те же правила блестят в одном проекте
и проваливаются в другом. Решает читаемость и связность, а не счётчик строк.

### 3.11. Доменное моделирование (DDD) ⚙️

Когда бизнес-логика размазана по сервисам и контроллерам, а модели — лишь мешки
полей, сложность растёт и баги становятся неуловимыми. **Domain-Driven Design**
переносит правила предметной области в саму модель.

- **Доменные примитивы** вместо «голых» `str`/`int`: тип `Email`, `Money`,
  `OrderId` валидирует себя в конструкторе — невалидное состояние нельзя собрать.
  Это убирает разбросанную по коду валидацию.
- **Объекты-значения (value objects)** — без идентичности, иммутабельны (§3.8),
  сравниваются по значению (`Money`, `DateRange`).
- **Сущности** — с идентичностью и жизненным циклом (`User`, `Order`).
- **Агрегаты** — группа объектов с единым корнем; менять внутренности можно
  только через корень, который держит инварианты.
- **Богатые vs анемичные модели:** поведение живёт в модели, а не в «сервисах».
  Анемичная модель (только геттеры/сеттеры) разрешает менять данные снаружи в
  обход правил — прямой путь к рассогласованному состоянию. **Не давай менять
  состояние напрямую — только через методы, хранящие инварианты.**
- **Ubiquitous language:** имена в коде = термины бизнеса; одно слово — одно
  значение (§3.2).
- **Bounded context:** явные границы, где термин имеет один смысл; они же —
  естественные границы сервисов (§9.9).

**Внедряй постепенно:** не переписывай всё разом. Начни с самого болезненного
поддомена — вынеси доменные примитивы и инварианты в богатую модель, остальное
оставь как есть.

## 4. Тестирование

### 4.1. Пирамида тестов ⚙️

```
      ╱╲
     ╱E2E ╲        ← мало, медленные (секунды-минуты)
    ╱──────╲
   ╱ Integr. ╲      ← средне (секунды)
  ╱────────────╲
 ╱  Unit tests  ╲    ← много, быстрые (миллисекунды)
╱────────────────╲
```

Современный сдвиг: **integration-тестов делают больше**, чем в классической
пирамиде («testing trophy») — они дают лучшее соотношение уверенности к цене.

### 4.2. Unit-тесты

Тестируют одну функцию/класс изолированно. Не трогают сеть/диск/БД, мокают
внешние зависимости, очень быстрые. Один тест = одна проверка.

### 4.3. Arrange-Act-Assert (AAA) + читаемость

```python
def test_add_punctuation_capitalizes_first_letter():
    # Arrange
    text = "привет мир"
    # Act
    result = add_punctuation(text)
    # Assert
    assert result == "Привет мир."
```

Имя теста = спецификация: читаешь имена тестов и понимаешь, как работает код.

### 4.4. Что и чем тестировать

- **Чистая логика** — unit-тесты (быстрые, много).
- **Связки модулей, БД, API** — integration-тесты (на реальной БД через
  Testcontainers, а не на хрупких моках).
- **Критические пользовательские сценарии** — E2E (Playwright для веба,
  по 1-2 на ключевой флоу, не больше).
- **Внешние HTTP** — мокай на сетевом уровне (MSW в TS, `respx`/`responses`
  в Python), а не подменяй свои функции.

### 4.5. Покрытие — инструмент, не цель

80% — разумный ориентир, 100% — обычно пустая трата сил. Важнее покрывать
**риск** (платежи, авторизация, граничные случаи), а не геттеры. Любой
найденный баг → сначала тест, воспроизводящий его, потом фикс (regression test).

### 4.6. Линтер и type checker — перед каждым коммитом

```bash
# Python
ruff check . && ruff format --check . && mypy .
# TS/JS
biome check .            # или: eslint . && prettier --check .
tsc --noEmit
# Go
go vet ./... && golangci-lint run
# Rust
cargo clippy -- -D warnings && cargo fmt --check
```

## 5. CI/CD (базовый уровень) ⚙️

### 5.1. Что гонять на каждый push/PR

```
1. Линтер           (ruff / biome / clippy)
2. Type checker     (mypy / tsc / —)
3. Тесты            (pytest / vitest / go test / cargo test)
4. Сборка/Build     (artefact компилируется/собирается)
5. Аудит зависимостей (pip-audit / osv-scanner / npm audit / cargo audit)
6. Секрет-сканер    (gitleaks)
```

### 5.2. Пример пайпа (GitHub Actions, Python+uv)

```yaml
name: CI
on: [push, pull_request]

jobs:
  ci:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: astral-sh/setup-uv@v5
        with:
          enable-cache: true        # кэш зависимостей встроен
      - run: uv sync --all-extras --dev
      - run: uv run ruff check .
      - run: uv run ruff format --check .
      - run: uv run mypy .
      - run: uv run pytest
```

> Подбирай `runs-on` под цель сборки: `ubuntu-latest` для веба/сервисов;
> `windows-latest` / `macos-latest` — если собираешь нативный desktop под эти ОС.

### 5.3. Принципы здорового пайпа

- **Кэшируй** зависимости (setup-uv/setup-node делают это из коробки) — пайп в
  2-5 раз быстрее.
- **Падай быстро:** дешёвые проверки (линтер) — раньше дорогих (E2E).
- **Воспроизводимость:** ставь зависимости из lock-файла (`uv.lock`,
  `pnpm-lock.yaml`, `Cargo.lock`), а не «последние».
- **Pin actions** по major-тегу (`@v4`) или SHA для безопасности.

### 5.4. Git Hooks (pre-commit)

Гоняй линтер/формат локально перед коммитом — не дай сломать main.

```yaml
# .pre-commit-config.yaml
repos:
  - repo: https://github.com/astral-sh/ruff-pre-commit
    rev: v0.6.0
    hooks: [{ id: ruff }, { id: ruff-format }]
  - repo: https://github.com/gitleaks/gitleaks
    rev: v8.18.0
    hooks: [{ id: gitleaks }]
```

В JS-проектах аналог — `lefthook` или `husky` + `lint-staged`.

### 5.5. Автоматический релиз

Тег `v*` → CI собирает артефакт и публикует. Версию и changelog можно
генерировать автоматически из Conventional Commits (`release-please`,
`changesets` для JS-монорепо, `semantic-release`).

```bash
git tag v1.2.3 && git push --tags
# → CI собирает exe/wheel/docker image и публикует в Releases/Registry
```

## 6. Безопасность (база) ⚙️

### 6.1. Секреты — только из окружения/секрет-менеджера

```ts
const apiKey = process.env.API_KEY;        // не хардкод
if (!apiKey) throw new Error("API_KEY is required");
```

В проде — секрет-менеджер (Vault, AWS/GCP Secrets Manager, Doppler,
Infisical), не `.env` на диске. Ключи — ротируемые, с минимальным сроком жизни.

### 6.2. Зависимости и supply-chain

- Регулярно сканируй: `pip-audit`, `osv-scanner`, `npm audit`, `cargo audit`.
- Автообновления: Dependabot / Renovate.
- Фиксируй версии lock-файлом; в проде — **воспроизводимая** установка.
- Проверяй integrity загрузок (SHA-256/подпись), особенно бинарников и моделей.
- Для релизов — генерируй SBOM (`syft`) и, по возможности, подписывай
  артефакты (Sigstore/`cosign`).

### 6.3. Least privilege

Сервису — только нужные права (read на свою таблицу, не root). CI-токену —
доступ только к своему репозиторию. Контейнеру — non-root, read-only FS где
можно.

### 6.4. Не пиши свою криптографию и аутентификацию

Бери проверенные библиотеки/провайдеры (см. §III). Хеши паролей — `argon2id`
или `bcrypt`, никогда не «самодельный» хеш. Валидируй и экранируй любой ввод.

### 6.5. OWASP Top 10 (2021) — минимум для всех

1. **Broken Access Control** — проверяй права на **каждый** запрос.
2. **Cryptographic Failures** — шифруй чувствительные данные (в покое и в
   транзите), не храни лишнего.
3. **Injection** (SQL/NoSQL/командная/XSS) — параметризуй запросы, экранируй
   вывод.
4. **Insecure Design** — закладывай угрозы на этапе проектирования.
5. **Security Misconfiguration** — дефолты, лишние порты, отладочные эндпоинты.
6. **Vulnerable Components** — см. §6.2.
7. **Identification & Auth Failures** — не катай свою аутентификацию.
8. **Software & Data Integrity** — supply-chain, подпись артефактов.

> Для **LLM-агентов** действует отдельный список **OWASP Top-10 for LLM (2025)**
> (prompt injection, excessive agency и др.) — не путать с OWASP 2021. См. §19.2.1.

### 6.6. Постквантовая криптография и crypto-agility 🌐

Угроза «**Harvest Now, Decrypt Later**»: трафик и БД крадут зашифрованными уже
сегодня, чтобы расшифровать, когда дозреет квантовый компьютер. Данные с долгим
сроком секретности (медицина, персональные данные, гостайна) под риском **сейчас**.

- **Crypto-agility:** не зашивай алгоритм намертво — держи его параметром, чтобы
  заменить без переписывания. Урок MD5/SHA-1 (коллизия 2004 → поддельный
  SSL-сертификат 2008): миграция занимает годы, начинать поздно — дорого.
- **Не используй сломанные примитивы:** MD5/SHA-1 — только для несекьюрных
  контрольных сумм, никогда для подписей и паролей (§6.4).
- **PQC-алгоритмы (NIST):** ML-KEM (Kyber) для обмена ключами, ML-DSA (Dilithium)/
  SLH-DSA для подписей. Переход — через **гибридные схемы** (классика + PQC).
- **Срок жизни секрета** должен быть короче времени, за которое его реально
  взломают; ротация ключей (§6.1) — часть защиты от HNDL.

## 8. Процессы и культура

### 8.1. Post-mortem без обвинений (blameless)

Упало — ищем **причину и решение**, не виноватого.
Плохо: «Вася задеплоил без тестов». Хорошо: «Не было pre-commit hook, не
дающего закоммитить без тестов → добавляем hook».

### 8.2. Ограничивай WIP

Не начинай новую фичу, не доведя и не замержив текущую. 3 незавершённых PR =
распыление и ничего не доведено до конца.

### 8.3. Code review = обучение

Видишь ошибку — объясни **почему** это ошибка. Предлагаешь другой подход —
объясни **почему** он лучше. «Давай вынесем в константу», а не «это ужас».

### 8.4. Технический долг

Иногда «быстро и криво» оправдано — но фиксируй долг (TODO с тикетом) и
возвращай. Scout rule: **оставляй код чище, чем нашёл.**

# Часть II. Специфика по типу проекта

## 9. Backend / сервис 🌐

### 9.1. Логирование → stdout/stderr

В контейнере/сервисе пиши логи **в stdout/stderr** структурированным JSON —
система сбора (journald, Loki, Datadog, CloudWatch) заберёт сама. Не пиши в
файлы вручную.

```python
# структурированный лог (structlog / std logging + JSON formatter)
logger.info("model_loaded", model_id=model_id, duration_ms=elapsed)
```

```
2026-06-17T10:30:00Z INFO  [worker] model_loaded model_id=vosk-ru duration_ms=420
2026-06-17T10:30:05Z ERROR [worker] transcribe_failed reason=timeout
```

Формат: **время + уровень + модуль + событие + контекст**. Добавляй `trace_id`
для сквозной трассировки.

### 9.2. Наблюдаемость (observability)

Три столпа — логи, метрики, трейсы — через **OpenTelemetry** (вендор-нейтрально).

- **Метрики RED:** Rate (запросы/с), Errors (доля ошибок), Duration (latency
  p50/p95/p99).
- **Метрики USE** для ресурсов: Utilisation, Saturation, Errors.
- Ошибки → Sentry/GlitchTip с группировкой и алертами.
- Для **AI-агентов** RED/USE недостаточно (`200 OK` скрывает провал) — добавь
  агент-метрики: tool success per-tool, cost-per-task, детекцию циклов (§19.2.3).

### 9.3. Алерты по симптомам, не по причинам

Хорошо: «p99 latency > 500ms 5 минут», «error rate > 1% за 10 минут», «сервис
не отвечает на health-check». Плохо: «CPU > 90%» (вечно горит → игнорируют),
«упал 1 запрос из миллиона» (шум).

### 9.4. SLO и error budget

Определи целевой уровень (например, 99.9% аптайм = ~8.7 ч/год даунтайма). Пока
budget не потрачен — можно рисковать с деплоем; кончился — только багфиксы.

### 9.5. Деплой

| Стратегия | Суть | Риск |
|-----------|------|------|
| Rolling | постепенно заменяем поды | низкий |
| Blue-green | два окружения, мгновенное переключение трафика | очень низкий |
| Canary | новая версия на 1-5% трафика, затем больше | минимальный |
| Recreate | стоп старой → старт новой | есть даунтайм |

- **Immutable tags:** тегируй образы по SHA коммита (`myapp:abc123`), никогда
  `:latest`.
- Миграции БД — обратимо-совместимые (expand/contract), отдельно от деплоя кода.

### 9.6. Graceful shutdown

По `SIGTERM`: перестать принимать новое → доработать текущее (таймаут ~30с) →
закрыть соединения (БД/файлы) → выйти.

```python
async def shutdown():
    server.stop_accepting()
    await drain_inflight(timeout=30)
    await db.close()
```

### 9.7. Health checks

`/livez` (жив ли процесс), `/readyz` (готов принимать трафик — БД/зависимости
подключены), `/metrics` (для Prometheus/OTel).

### 9.8. Защита эндпоинтов

Rate limiting на публичных ручках, размер тела запроса, таймауты, CORS по
whitelist, идемпотентность для платежей, валидация входа схемой.

```python
@app.get("/api/search")
@limiter.limit("100/minute")
async def search(...): ...
```

### 9.9. Выбор архитектурного стиля: монолит ↔ микросервисы 🌐

Споры «монолит vs микросервисы» напоминают религиозные. На практике **серебряной
пули нет**, а оптимум обычно посередине.

- **Начинай с модульного монолита.** Чёткие модули внутри одного развёртывания
  дают почти всю выгоду границ без сетевой боли. Дроби на сервисы, только когда
  упёрся в реальный предел (масштабирование части, независимые релизы команд).
- **Граница сервиса = bounded context** (§3.11), а не «по таблицам». Неверные
  границы рождают **распределённый монолит** — худшее из обоих миров: сетевые
  вызовы + жёсткая связанность + каскадные отказы.
- **Известные кейсы:** Amazon Prime Video сократил расходы ~на 90%, вернувшись к
  монолиту для одного сервиса; Uber боролся со «звездой смерти» из зависимостей;
  крупные компании пересобирают раздутые микросервисы обратно.
- **Saga / event-driven / CQRS** — инструменты под конкретную задачу
  (распределённые транзакции, развязка, разделение чтения/записи), а не цель сама
  по себе.
- **Цена распределённости:** сетевые отказы, согласованность, идемпотентность
  (§9.8), сквозной трейсинг (§9.2). Бери её осознанно.

**Готов ли проект к микросервисам?**
- [ ] Команды реально независимы и упираются в общий релиз-цикл?
- [ ] Есть зрелые observability, CI/CD и автоматизация инфры (§5, §9.2)?
- [ ] Границы доменов понятны и стабильны (§3.11)?
- [ ] Части системы нужно масштабировать/релизить независимо?

Если на большинство «нет» — модульный монолит сейчас выгоднее.

## 10. Frontend / веб 🎨

### 10.1. Производительность — Core Web Vitals

Цели: **LCP < 2.5s**, **INP < 200ms**, **CLS < 0.1**. Инструменты: Lighthouse
CI в пайпе, бюджет бандла (size-limit).

- Code-splitting и lazy-load по маршрутам; грузи только нужное.
- Оптимизируй изображения (AVIF/WebP, `srcset`, lazy-loading).
- Минимизируй клиентский JS: предпочитай server components / SSR / островную
  гидрацию там, где можно.
- Избегай лишних ре-рендеров (мемоизация, стабильные ключи/коллбэки).

### 10.2. Состояние

- **Серверное состояние** (данные с API) — через TanStack Query / RTK Query
  (кэш, инвалидация, ретраи), **не** руками в глобальном сторе.
- **Клиентское UI-состояние** — локально или лёгкий стор (Zustand/Jotai);
  Redux только если реально нужен.
- Формы + валидация — React Hook Form + Zod.

### 10.3. Доступность (a11y)

Семантический HTML, alt у изображений, фокус-стейты, навигация с клавиатуры,
контраст. Проверяй `axe`/Lighthouse a11y в CI. Это не опция, а требование.

### 10.4. Безопасность фронта

- **CSP** (Content-Security-Policy) против XSS; никогда `dangerouslySetInnerHTML`
  без санитайза (DOMPurify).
- Токены — в httpOnly-cookie, **не** в `localStorage`.
- Не клади секреты в клиентский бандл (всё в нём — публично).

### 10.5. UX-мелочи, которые ждут

Состояния loading/empty/error для каждого асинхронного экрана, оптимистичные
апдейты, debounce ввода, отмена устаревших запросов.

## 11. Desktop-приложения 🖥

### 11.1. Логирование → файл с ротацией (а НЕ stdout)

У оконного (windowless) `.exe`/бандла **нет консоли** — stdout некуда писать.
Логи должны идти в файл в пользовательской директории с ротацией.

```python
from logging.handlers import RotatingFileHandler
handler = RotatingFileHandler(
    appdata_dir() / "logs" / "app.log",
    maxBytes=1_000_000, backupCount=3, encoding="utf-8",
)
```

- Путь — стандартный для ОС: `%APPDATA%` (Windows), `~/Library/Logs` (macOS),
  `~/.local/state` (Linux/XDG).
- Дай пользователю пункт меню «Открыть логи».
- **Не** пиши в лог чувствительные данные (пароли, seed-фразы, токены).

### 11.2. Сборка, подпись, обновления

- **Подписывай билды:** code signing (Windows Authenticode / Apple Developer ID
  + notarization) — иначе SmartScreen/Gatekeeper будут пугать пользователя.
- Публикуй **SHA-256** релиза; давай возможность собрать из исходников.
- Автообновления — встроенный апдейтер (Tauri updater, Squirrel/electron-
  updater) **с проверкой подписи** обновления.
- Минимизируй права/доступы; в Tauri — строгий allowlist возможностей.

### 11.3. Прочее

Не блокируй UI-поток (тяжёлое — в воркер/поток). Корректно обрабатывай
оффлайн-режим и отсутствие прав на запись.

| AI-агент: guardrails | классификаторы prompt-injection (вход), PII-фильтры (выход), allowlist на tools — вне агента (§19.2.1) |
| AI-агент: LLM-наблюдаемость | LLM-трейсинг (Langfuse, Helicone, Phoenix, LangSmith) или OTel поверх Grafana; проверяй актуальное |

### 19.1. AI-ассистированная разработка ⚙️

AI-кодинг-ассистенты (Cursor, Claude Code, VS Code/Copilot, Windsurf,
Gemini CLI и т.п.) кратно ускоряют разработку — но только при правильной
дисциплине. Без неё агент генерирует код, который выглядит правильно, но не
работает, не тестирован и не соответствует замыслу.

> **Ключевая мысль:** LLM по умолчанию оптимизирует на «выглядит правильно»,
> а не на «работает правильно». Правила ниже переключают поведение на
> инженерную дисциплину.

#### 19.1.1. Главные проблемы AI-кодинга

Прежде чем давать правила — вот проблемы, которые они решают:

| Категория | Проблема | Суть |
|-----------|---------|------|
| Процесс | Код без спека | Нет дизайна → агент галлюцинирует архитектуру |
| Процесс | Scope creep | Один промпт = 500 строк, невозможно проверить |
| Процесс | Blind acceptance | Программист принимает код, который не понимает |
| Процесс | Нет инкрементальной проверки | Огромный diff без промежуточных тестов |
| Качество | Тавтологические тесты | Тест повторяет имплементацию, а не поведение |
| Качество | Happy-path only | Агент пропускает обработку ошибок / краевые случаи |
| Качество | Over-engineering | Лишние абстракции, паттерны ради паттернов |
| Качество | Некогерентность с проектом | Свой стиль/паттерны вместо существующих |
| Качество | Небезопасный код | Хардкод секретов, SQL-инъекции, небезопасные дефолты |
| Контекст | Галлюцинация API | Несуществующие/устаревшие функции, методы, флаги |
| Контекст | Галлюцинация зависимостей | Импорт пакета, которого не существует |
| Контекст | Context pollution | Шум в контексте снижает качество ответов |
| Контекст | Context drift | Деградация в длинных сессиях |
| Контекст | Незнание проекта | Агент не знает конвенций, структуру, стек проекта |
| Мета | Деквалификация | Программист перестаёт понимать свой код |
| Мета | Ложная уверенность | «Агент написал — значит правильно» |

#### 19.1.2. Лестница лени (минимальное решение)

> **Принцип:** лучший код — тот, который не написан. «Ленивый» значит
> эффективный, а не небрежный. Лень касается *решения*, никогда —
> *понимания задачи*: сперва прочитай код, который меняешь, и проследи
> реальный поток end-to-end, и только потом выбирай ступень.

Перед тем как писать код, остановись на **первой подходящей ступени**:

1. **Это вообще нужно?** Спекулятивная потребность → пропусти, скажи одной
   строкой почему. (YAGNI)
2. **Уже есть в этом проекте?** Хелпер, утилита, тип, паттерн рядом →
   переиспользуй, не переписывай. Посмотри, прежде чем писать — повторная
   реализация того, что лежит в паре файлов, самый частый слоп.
3. **Stdlib умеет?** Используй.
4. **Нативная фича платформы покрывает?** `<input type="date">` вместо
   библиотеки-пикера, CSS вместо JS, ограничение БД вместо кода приложения.
5. **Уже установленная зависимость решает?** Используй. Не добавляй новую
   ради того, что делается парой строк.
6. **Можно одной строкой?** Одной строкой.
7. **И только тогда:** минимум кода, который работает.

Две ступени подходят → бери верхнюю и иди дальше. Два варианта stdlib
одного размера → бери тот, что корректен на краевых случаях: лень — это
меньше кода, а не более хлипкий алгоритм.

**Баг-фикс = корневая причина, а не симптом.** Тикет называет симптом.
Перед правкой — `grep` всех вызывающих функции, которую трогаешь. Ленивый
фикс И ЕСТЬ корневой: одна проверка в общей функции — меньший diff, чем
проверка в каждом вызывающем, и не оставляет сломанными «соседние» вызовы.
Чини один раз там, где сходятся все вызовы.

**Что НИКОГДА не упрощать (вне торга):** валидация на границах доверия,
обработка ошибок, предотвращающая потерю данных, безопасность, базовая
доступность (a11y), всё явно запрошенное. Ленивый код без своей проверки —
незавершён: нетривиальная логика (ветвление, цикл, парсер, денежный или
секьюрити-путь) оставляет ОДНУ запускаемую проверку — наименьшее, что
падает, если логика сломана.

**Помечай сознательные упрощения комментарием-тегом.** Каждый намеренный
shortcut с известным «потолком» получает комментарий, называющий потолок
*и* путь апгрейда:

```python
# debt: глобальный лок; пер-аккаунт локи, если упрётся пропускная способность
# debt: O(n²) скан; индекс, если список перевалит за ~1k
```

Единый префикс (`debt:` / `shortcut:`) позволяет потом собрать их грепом
(`grep -rnE '(#|//) ?debt:' .`) в ledger техдолга (см. §8.4), чтобы «потом»
не стало «никогда». Комментарий без пути апгрейда — кандидат на тихое
гниение, помечай такие отдельно.

#### 19.1.3. Правила для AI-агента (инструкция модели)

Правила ниже можно скопировать в `RULES.md` / `.cursorrules` / системный
промпт любого AI-клиента. Они написаны как инструкция *к модели*.

---

**Роль.** Ты — исполнитель с правом голоса, а не автор проекта. Человек
определяет *что* строить. Ты предлагаешь *как*, но не принимаешь
архитектурных решений молча. Если видишь проблему — скажи, но не
переписывай дизайн самовольно.

**Правила процесса:**

1. **Уточни задачу.** Прежде чем писать код — убедись, что понимаешь *что*,
   *почему* и *ограничения*. Если что-то неясно — спроси.
2. **Сначала спек, потом код.** Опиши решение текстом (модули, интерфейсы,
   поток данных), согласуй с человеком, зафиксируй дизайн. Запрещено
   генерировать 500 строк по однострочному промпту.
3. **Мелкие шаги.** Каждый шаг = одно логическое изменение с проверяемым
   результатом. Не делай следующий, пока текущий не проверен.
4. **Ревью.** После завершения пройдись по коду: логика, краевые случаи,
   безопасность, производительность, стиль проекта.

**Правила кода:**

5. **Следуй проекту, а не себе.** Перед написанием — изучи структуру,
   стиль именования, паттерны, библиотеки. Новый код должен выглядеть так,
   будто его написал тот же человек.
6. **Переиспользуй.** Перед созданием утилиты/хелпера — проверь, нет ли
   аналога в проекте.
7. **Минимальный diff.** Не ставь библиотеки, о которых не просили. Не
   создавай абстракции «на будущее». Не меняй код за пределами задачи.
   Не рефактори «заодно».
8. **Не хардкодь.** Секреты — из env, магические числа — в константы,
   пути — через стандартные API.
9. **Обрабатывай ошибки.** Каждый внешний вызов = потенциальная ошибка.
   Лови конкретные исключения. Сообщения — с контекстом.
10. **Строгая типизация.** Никогда `any` — используй `unknown` + сужение.
    Данные «с границы» — валидируй схемой (Zod, Pydantic).

**Правила тестирования (TDD):**

11. **Нет продакшен-кода без падающего теста.** Напиши тест → убедись что
    падает (RED) → минимальный код чтобы прошёл (GREEN) → рефактори. Если
    код написан до теста — удали, напиши тест, потом имплементацию.
12. **Тест проверяет поведение, не имплементацию.** Тест с конкретным
    ожидаемым значением из спека, не с копией формулы из кода.

    ```python
    # ПЛОХО — тавтологический тест
    assert calculate(2, 3) == 2 * 3 + 1  # копия формулы

    # ХОРОШО — ожидание из спека
    assert calculate(2, 3) == 7
    ```

13. **Краевые случаи обязательны.** Для каждой функции: пустой ввод,
    null, граничные значения, невалидный тип, ошибки зависимостей.
14. **Один тест — одна проверка.** Имя теста = спецификация:
    `test_register_user_rejects_duplicate_email`.
15. **Не мокай свой код.** Мокай внешние сервисы (HTTP, SMTP). Если нужен
    мок внутренней функции — рефактори архитектуру. Для БД —
    Testcontainers, не in-memory заглушки.

**Правила безопасности:**

16. **Параметризуй запросы.** Никогда f-string в SQL/shell/шаблонах.
17. **Никогда не хардкодь секреты** — даже «для примера», даже «временно».
18. **Валидируй всё извне.** Любой пользовательский ввод — враждебный.
19. **Минимальные права.** Не проси прав, которые не нужны для задачи.
20. **Проверяй зависимости.** Пакет существует? Имя правильное (не
    typosquatting)? Указана конкретная версия?

**Правила контекста:**

21. **Не выдумывай API.** Если не уверен в сигнатуре/методе — проверь
    (прочитай файл, доку, исходник). Не угадывай.
22. **Не выдумывай зависимости.** Перед import — убедись: пакет существует,
    установлен в проекте, импортируемый модуль есть в этом пакете.
23. **Проверяй версии.** API меняется между версиями — используй API
    той версии, что стоит в проекте.
24. **Читай перед изменением.** Не предполагай содержимое файла по имени.
25. **Сообщай о неуверенности.** Не выдавай догадку за факт.

**Правила коммуникации:**

26. **Объясняй нетривиальные решения** — *почему* так, а не иначе.
27. **Показывай что изменил** — не заставляй искать отличия.
28. **Признавай ошибки** — скажи прямо, объясни, предложи исправление.
29. **Не защищай простоту прозой.** Если объяснение длиннее кода — удали
    объяснение: каждый абзац, оправдывающий упрощение, — это сложность,
    протащенная обратно в виде текста. (Явно запрошенное объяснение —
    отчёт, разбор, пошаговые заметки — давай полностью, это не долг.)

#### 19.1.4. Таблица антипаттернов

| # | Антипаттерн | Почему опасно |
|---|-----------|-------------|
| 1 | Код без согласованного дизайна | Результат не соответствует замыслу |
| 2 | Тесты *после* кода, повторяющие логику | Тесты бесполезны — пропускают баги |
| 3 | Вызов API/функции по памяти | Галлюцинация → runtime-ошибка |
| 4 | Импорт несуществующего пакета | Код не запустится |
| 5 | Пропуск обработки ошибок | Happy-path ≠ продакшен |
| 6 | Изменения за пределами задачи | Сайд-эффекты, сломанный diff |
| 7 | Абстракции «на будущее» | Over-engineering без пользы |
| 8 | Хардкод секретов / путей / конфига | Безопасность, переносимость |
| 9 | Игнорирование стиля проекта | Некогерентная кодовая база |
| 10 | Огромный блок кода без проверок | Невозможно ревьюить и отлаживать |
| 11 | Молчаливые архитектурные решения | Человек теряет контроль |
| 12 | Догадка вместо факта | Ложная уверенность → баги |
| 13 | Решение сложнее необходимого (не пройдена «лестница лени») | Over-engineering, лишний код и зависимости |
| 14 | Фикс симптома вместо корневой причины | Соседние вызовы остаются сломанными |
| 15 | Новая зависимость ради того, что есть в stdlib / платформе | Раздувание, риски supply-chain |
| 16 | Проза, защищающая упрощение, длиннее самого кода | Сложность, протащенная обратно текстом |

#### 19.1.5. Чеклист AI-агента перед каждым ответом

- [ ] Задача понятна? Если нет — спросил?
- [ ] Дизайн согласован?
- [ ] Код разбит на мелкие шаги?
- [ ] Тесты написаны *до* имплементации?
- [ ] Тесты проверяют *поведение*, не *имплементацию*?
- [ ] Краевые случаи обработаны?
- [ ] Все API/библиотеки — проверены, а не по памяти?
- [ ] Зависимости существуют и установлены?
- [ ] Секреты из env, не захардкожены?
- [ ] Стиль соответствует проекту?
- [ ] Diff минимален?
- [ ] Нетривиальные решения объяснены?
- [ ] Пройдена «лестница лени» (нет ли решения проще: stdlib / нативное / одна строка)?
- [ ] Фикс бьёт в корневую причину, а не в симптом (проверены все вызывающие)?
- [ ] Сознательные упрощения помечены тегом с потолком и путём апгрейда?
- [ ] Не упрощены валидация / обработка ошибок / безопасность / a11y?

#### 19.1.6. Оптимизация контекстного окна

Контекст LLM — конечный и дорогой. Чем больше шума, тем хуже качество
и выше расход токенов.

- **Фильтруй вывод команд.** Используй инструменты компрессии (например,
  [RTK](https://github.com/rtk-ai/rtk) — CLI-прокси, сокращающий токены
  на 60–90%) или передавай агенту только релевантные фрагменты.
- **Не кидай весь файл — кидай контекст.** Покажи функцию + тесты +
  зависимый интерфейс, а не 2000 строк целиком.
- **Компактные промпты.** «Добавь валидацию email в `register_user()`»
  лучше, чем «улучши регистрацию».
- **Чисти контекст.** В длинных сессиях начинай новый разговор —
  «забытый» контекст из начала уже деградировал.

#### 19.1.7. Параллельные AI-агенты и worktrees

Для крупных задач — несколько агентов параллельно, каждый в своём
git worktree (см. §2.4):

```bash
git worktree add ../proj-api feat/new-api    # Агент 1 — API
git worktree add ../proj-ui  feat/new-ui     # Агент 2 — UI
```

- Подзадачи должны быть **независимы** (разные модули/файлы).
- Каждый агент получает только **свою** часть плана.
- Результат каждого — через ревью перед merge.

#### 19.1.8. MCP и актуальная документация

**MCP (Model Context Protocol)** подключает к AI-ассистенту внешние
источники — документацию, GitHub, БД, логи.

**Context7** (MCP-сервер) подтягивает **актуальную, версионно-привязанную
документацию** библиотеки прямо в момент написания кода, снижая
галлюцинации по устаревшим API.

- **Подключение:** добавь в конфиг MCP; вызывай через «use context7» или
  `resolve-library-id` + `get-library-docs`.
- **Когда полезен:** быстро меняющиеся экосистемы (Next.js, Tailwind,
  Drizzle, FastAPI) и новые мажорные версии.

Другие MCP-серверы: GitHub, БД (read-only), логи. Давай **минимальные
права** (§6.3), не подключай непроверенные серверы.

> **Теневые (shadow) MCP-серверы** — серьёзный риск: сотрудники ставят серверы в
> обход IT, а каждый сервер просит широкие права (Gmail — полный доступ к почте,
> файловый — к домашней папке). Злоумышленники маскируют вредоносные серверы под
> полезные. Правила: ставь только из доверенных источников, давай минимальные
> права, держи реестр разрешённых серверов и проверяй их по supply-chain
> чек-листу (§19.2.6).

#### 19.1.9. Границы AI-ассистента

AI-ассистент — **не** замена:

- **Lock-файлов** — источник истины по версиям: `uv.lock` /
  `pnpm-lock.yaml` / `Cargo.lock`.
- **Аудита безопасности** — `pip-audit` / `osv-scanner` / `npm audit`
  обязательны (§6.2).
- **Code review** — AI-код требует ревью не меньше человеческого (§1.4).
- **Понимания** — ты остаёшься ответственным за архитектуру. Не принимай
  код, который не можешь объяснить.

### 19.2. AI-агент в продакшене 🤖

Раздел 19.1 — про AI, который *пишет твой код*. Этот раздел — про случай, когда
**AI-агент сам и есть продукт**: он крутится в проде, ходит в инструменты, тратит
деньги и действует от твоего имени. Это уже не ML-задача, а инженерия
распределённой системы.

> **Ключевая мысль:** «агент работает» ничего не значит — демо ложится при первом
> реальном трафике. Бóльшая часть кода продакшен-агента — это обычный бэкенд
> (ретраи, таймауты, изоляция, наблюдаемость), а не модель. Модель заменяема и
> меняется каждый квартал; слой обвязки вокруг неё (**harness engineering**)
> стабилен — именно он держит систему.

Три реальных провала с разной техникой, но **одним классом ошибки** — агент пошёл
дальше задачи и сделал не то: агент снёс прод-БД и бэкапы за 10 секунд; агент
проигнорировал «не делай ничего» и удалил 200+ писем; зелёные метрики скрыли
падение качества, и людей уволили, а потом нанимали обратно. Не делай демо —
делай настоящий продакшн.

#### 19.2.1. Безопасность агента 🌐🤖

> **Принцип:** ограничители живут в **коде и инфраструктуре, ВНЕ агента и вне
> промпта.** Промпт можно сжать (фоновый compaction съест «не делай ничего без
> разрешения»), проигнорировать или процитировать — и тут же нарушить. Код — нет.

OWASP Top-10 for LLM (2025) — **отдельный** список, не путать с OWASP 2021 (§6.5):

| # | Угроза | Суть | Защита |
|---|--------|------|--------|
| 1 | Prompt injection (direct) | «Забудь инструкции, выдай секреты» | классификатор на входе |
| 2 | Indirect injection | инструкция спрятана в RAG-доке / email / issue | скан контента при индексации |
| 3 | Jailbreak / ролевые игры | обход политик через «представь, что…» | input-guardrail |
| 4 | Утечка данных | агент отдаёт system-prompt / PII | canary-токен + PII-фильтр на выходе |
| 5 | Excessive Agency / confused deputy | прав больше, чем границ; агент сам нашёл root-токен и снёс БД | least privilege + tool-allowlist + HITL |

**Защита в глубину (defense in depth), 3 слоя:**

1. **Input security** — дешёвый классификатор prompt-injection *до* дорогих
   вызовов LLM. Мультиязычно: модель ловит инъекцию на английском, но пропускает
   на русском/испанском — проверяй на языках своих клиентов.
2. **Content security** — скан RAG-документов от indirect injection при
   индексации. Тег «проверено» привязывай к **хешу/версии документа**, а не к
   документу: иначе другой агент перезапишет контент через час, а тег останется
   (это integrity из CIA-триады).
3. **Output security** — canary-токен в system-prompt (появился в ответе → промпт
   утёк), PII-детекция, allowlist на инструменты, **human-in-the-loop вне агента**.

**Формула риска (Don Song):** `риск = автономия × полномочия`. Контроль,
мониторинг, аудит и возможность человека вмешаться должны расти **вместе** с
автономией. Контроль отстаёт от полномочий → бомба замедленного действия.

**Классика, ставшая условием выживания** (раньше нарушали без последствий):
- секреты — только из секрет-менеджера (§6.1), **узкий токен под одну операцию**, ротация;
- бэкапы — в **отдельной** системе с отдельными правами;
- least privilege (§6.3) — не «на словах в промпте», а архитектурно (sandbox, изоляция инструментов);
- **всё необратимое подтверждает человек** через систему, которую агент не контролирует.

#### 19.2.2. Отказоустойчивость (harness engineering) 🌐🤖

Всё, что ты знаешь про распределённые системы, нужно агенту один-в-один:
**ретраи с jitter, circuit breaker, таймауты, идемпотентность tool-вызовов,
очереди для медленных задач, backpressure, graceful degradation** (§9.5).

Провайдеры падают (AWS, Anthropic `529 Overloaded`, OpenAI `503`, `429`).
Пользователь винит **тебя**, а не Amazon/Anthropic. Что делать при падении
зависимости — **продуктовое решение** (PM + безопасник + юрист), а не выбор
разработчика за клавиатурой, и принимается **по каждой фиче отдельно**:

| Стратегия | Когда | Пример |
|-----------|-------|--------|
| **Fail closed** | безопасность критична: банки, медицина, платежи | классификатор упал → блок. Лучше ничего, чем неправильно |
| **Fail open** | read-only, потеря юзера дороже риска | guardrail упал → пропускаем. Лучше риск, чем блок живого клиента |
| **Graceful degradation** | большинство реальных случаев | основной провайдер → резервный → кэш → простой шаблон / эскалация оператору |

Пример e-commerce: checkout = fail closed; поиск/витрина = degradation
(Elasticsearch вместо LLM, топ продаж вместо пустой страницы). Решение — на этапе
проектирования фичи, не в момент написания кода.

#### 19.2.3. Наблюдаемость агента 🌐🤖

> **Принцип:** полностью наблюдаемой системы не существует. Observability — не
> «настроил и забыл», а **непрерывный процесс**: каждый инцидент/инсайт вскрывает
> слепое пятно → добавляешь метрику. Зрелость — в дисциплине, а не в трёх
> «golden»-метриках на старте.

Стандартного RED/USE (§9.2) **мало**: финальный статус `200 OK`, а агент сделал не
то, зациклился или сжёг бюджет. Нужны **агент-специфичные** метрики (их обычно не
ставят):

- **Tool-call success rate — с разбивкой ПО каждому инструменту** + p95 latency на
  тул. Инструмент может неделями возвращать не ошибку, а пустой результат.
- **Cost per resolved task** (не per LLM-call) — сумма стоимости всех вызовов в
  рамках одной завершённой задачи. Нужна концепция сессии/задачи. Самая важная и
  самая пропускаемая метрика агентов.
- **Token counting с учётом reasoning-токенов** — биллинг считает рассуждения, а
  наивная метрика «токены ответа» врёт (расхождение бывает в 40×).
- **Context utilization / частота compaction** — высокая = риск потери критичных
  инструкций и падения качества; объясняет, почему две одинаковые сессии ведут
  себя по-разному (в одной compaction сработал, в другой нет).
- **Детекция «плохих циклов»** — семантический повтор tool-запросов (агент крутит
  один и тот же поиск с минорными вариациями, каждый раз `200`). Метрика —
  дистанция в embedding-пространстве между последовательными действиями; ниже
  порога → цикл.
- **Качество выбора инструмента** — пошёл в поиск по почте вместо календаря.
  Оценивается отдельной judge-моделью / на синтетике из реальных запросов.
- **Трейсинг по стадиям пайплайна** (классификатор → роутер → RAG → tools →
  генерация → security → HITL). Без него агент — чёрный ящик.

#### 19.2.4. Выбор LLM в продакшене 🌐🤖

В проде работает не одна модель, а **связка**: основная генерация + маленький
роутер (классифицирует запрос за миллисекунды) + эмбеддинги для RAG + реранкер +
security-модели + vision + **fallback** на случай падения основного провайдера.

> **Бенчмарки (MMLU, HumanEval, SWE-bench) почти бесполезны для выбора.** Они
> отсеивают совсем слабых, но не говорят, какая модель лучше **на твоём домене**.
> Тестируй кандидатов **на своих данных и реальных сценариях** (десятки кейсов, не
> тысячи: основные паттерны трафика + пара сложных мульти-tool). Один и тот же
> tool-description ведёт себя по-разному на GPT/Gemini/Qwen — fallback-модель и
> роутер тестируй так же, как основную.

**«Бесплатных моделей не бывает.» Считай 4 статьи затрат, а не «GPU vs API»:**

1. **API-доллары** (по токенам);
2. **инфраструктура** (GPU / managed inference);
3. **DevOps-время** (поднять, обновлять, чинить — присутствует всегда; зарплата SRE часто > стоимости железа);
4. **«доллары сложности»** (cold start, тензор-параллелизм, NVIDIA блокирует P2P на потребительских картах).

Дерево решения: регуляторика / data residency → масштаб и экономика (посчитать
счёт на текущей нагрузке и ×3) → нужен ли fallback/routing →
**API vs managed cloud GPU vs self-host**. Лозунги «API — рабство» / «open source —
для серьёзных» — это не инженерия.

#### 19.2.5. Экономика контекста и кэш ⚙️🤖

Понимание устройства LLM напрямую бьёт по бюджету:

- **Prefill (обработка промпта) — параллельный и дешёвый; decode (генерация) —
  последовательный и дорогой** (output ~в 3–5× дороже input). Держи ответы компактными.
- **Prompt caching** — запись в кэш ~на 25% дороже обычного input, зато чтение
  **−90%**. На повторяющемся system-prompt/документации экономит 70–90% бюджета
  (TTL обычно ~5 мин). **Batch API** для несрочных задач (суммаризация,
  классификация) — ещё −50%.
- **Контекст ≠ свалка.** Context engineering важнее prompt engineering: промпт —
  лишь триггер, результат определяет контекст (память + retrieval + состояние +
  инструменты + сборка). Инструменты съедают контекст агрессивно: один tool-вызов
  = тысячи токенов.
- **Русский текст в токенах в 1.5–2× дороже английского** — закладывай в расчёт
  размера окна и стоимости.

#### 19.2.6. Происхождение моделей (supply chain) 🌐🤖

Если берёшь модель с Hugging Face / дообучаешь — это вектор атаки. Исследование
Anthropic: **~250 отравленных документов бэкдорят модель любого размера**
(одинаково для 600M и 13B), процент от датасета не важен; fine-tuning с 1%
отравленных данных создаёт backdoor.

**Чек-лист перед использованием модели:**
- [ ] Издатель на HF проверенный?
- [ ] Больше тысячи скачиваний?
- [ ] Есть карточка модели с документацией обучения?
- [ ] Просканирована (ModelScan и аналоги)?
- [ ] Протестирована в песочнице (изолированно), а не сразу в проде?

> **Shadow AI** — 73% корпоративного использования ChatGPT идёт через личные
> аккаунты, 20% организаций словили утечку. Нужна явная политика: что и куда можно
> загружать.

#### 19.2.7. Таблица антипаттернов (агент в проде)

| # | Антипаттерн | Почему опасно |
|---|-------------|---------------|
| 1 | Ограничители в промпте, а не в коде | compaction/инъекция снимут их на лету |
| 2 | Широкие права / root-токен «чтобы работало» | excessive agency → агент снесёт прод |
| 3 | Бэкапы рядом с данными, общий токен | удаляются вместе с данными |
| 4 | Нет fallback при падении провайдера | юзер винит тебя, не Anthropic |
| 5 | Стратегия фолбэка выбрана разработчиком, не продуктом | fail-open там, где нужен fail-closed |
| 6 | RED / `200 OK` как единственная правда | скрывает плохой ответ, циклы, слив бюджета |
| 7 | Нет cost-per-task и трейсинга по стадиям | непонятно, где утекают деньги |
| 8 | Выбор модели по бенчмаркам без теста на своих данных | хуже качество в твоём домене |
| 9 | Модель с HF без скана и песочницы | backdoor / supply-chain |
| 10 | Нет HITL на необратимых действиях | один вызов сносит компанию |

#### 19.2.8. Чек-лист агента перед продом

- [ ] Ограничители (allowlist, лимиты, деструктивные операции) — в коде/инфре, вне промпта?
- [ ] Least privilege: узкие токены, бэкапы изолированы, sandbox инструментов?
- [ ] Input/content/output guardrails настроены и протестированы (в т.ч. на нужных языках)?
- [ ] HITL на всех необратимых действиях, вне агента?

- [ ] Стратегия при падении провайдера (closed/open/degradation) выбрана продуктом по каждой фиче?
- [ ] Fallback-модель и роутер протестированы на своих данных?
- [ ] Метрики: tool success per-tool, cost-per-task, циклы, context utilization, reasoning-токены?
